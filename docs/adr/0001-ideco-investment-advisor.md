# ADR 0001: iDeCo投資判断アドバイザー機能の追加

## Status

Proposed

## Context

利用者はSBI証券のiDeCo（セレクトプラン）で投資信託を運用しており、保有商品と乗り換え候補について「買い」「売り」「ホールド」の判断材料を定期的にSlackで受け取りたい。
実際の売買は利用者自身が行い、botの役割は投資判断情報の提供に限る。
ただし3値の判定はあいまいにせず明示する。

要件は次のとおり。

- 対象は保有商品に加え、カテゴリ代表（先進国株、全世界株、国内株、債券、金など）の乗り換え候補数本とし、登録制で管理する
- 判断根拠は基準価額の推移（定量）と市況ニュースやマクロ情勢（定性）の両方とする
- 米ドル/円の為替レートの現在値と変動見通しを判断と投稿に含める。為替ヘッジのない海外資産ファンド（全世界株式など）では、為替が基準価額変動のベース要因になるためである
- 判断履歴を保存し、前回判断からの変化とその理由を明示する。過去の推奨が当たったかの成績検証も毎週振り返る
- 投稿は週1回を基本とするが、間隔は設定変更のみで1日単位まで短縮できるようにする
- 同一日に何回実行しても投稿は1回に保つ（冪等性）。投稿済みかどうかはSlackの履歴を読んで判断する
- iDeCoの売買には日数がかかる（スイッチングは指示から完了まで概ね1週間から10日、掛金の配分変更は翌月拠出分から反映）ため、短期の値動きを狙う判断は成立しない。この特性を判断プロンプトに組み込む

## Decision

### 実装先: 本リポジトリに「advisor」系統を追加する

既存のdigestフローとは独立した実行系統として、本リポジトリ内に実装する。
Lambda、Terraform、Slack投稿、デプロイ基盤、agent-first設計（機能はagentのツールとプロンプトで実現する方針）をすべて再利用できるためである。

別リポジトリへの分離は、Lambda、Terraform、CI、Slack通知の複製が個人運用に対して過剰なため採らない。
既存sources機構への相乗り（プロンプトのみで実現）は、判断履歴の構造化保存と成績検証ができないため採らない。

### 起動と実行フロー

EventBridgeに日次ルール（cron: 毎日JST 17:00、input: `{"job": "advisor"}`）を追加する。
既存の日次JST 9:00ルールはdigest専用のまま変更しない。
`handler.py`はeventの`job`フィールドでdigest/advisorを分岐する。

advisorの実行手順は次のとおり。
`advisors`テーブルを全件scanし、アドバイザー単位で以下を実行する（digestのソース単位ループと同型）。

1. Slack履歴から当該アドバイザーの前回投稿（親メッセージ）の時刻を取得する。同一チャンネルにdigestや他のアドバイザーも投稿しうるため、当該アドバイザーの`title`をheaderに持つbot投稿で識別する
2. 前回投稿が同日（JST）なら投稿しない。前回投稿から`interval_days`日未満でも投稿しない。これにより1日に何回実行しても投稿は1回であり、間隔の短縮は`interval_days`の変更だけで済む
3. `products`の商品ごとに基準価額の時系列を取得し、`advisor-judgments`の過去判断と突き合わせて成績（判断時点の基準価額に対する現在の騰落率）をPythonで計算する
4. `agent.run_advice`が全商品の判断（BUY/SELL/HOLD）と理由を構造化出力で返す
5. 親メッセージ（市況サマリ、判断変更点、成績サマリ、免責）を投稿し、商品ごとの判断をスレッド返信する
6. 今回の判断を`advisor-judgments`に保存する

### ストレージ: DynamoDBに2テーブルを追加する

digestの`sources`が「1ソース = 1スレッド = 1アイテム」であるのと同型に、advisorも「1アドバイザー = 1スレッド = 1アイテム」とする。
別のアドバイザー（課税口座向けなど）を将来追加する場合も、アイテムを1件追加するだけで済み、DynamoDBで移行を伴うキー設計の変更が発生しない。

**`advisors`**：アドバイザー定義。
PKは`advisor_id`（例: `ideco-sbi`）。
属性は次のとおり。

- `channel_id`（投稿先）、`interval_days`（投稿間隔、既定7）、`title`（ヘッドラインheader兼投稿識別子。同一チャンネル内で一意にする）
- `products`（判断対象の商品リスト。各要素は`{isin, name, category, holding, assoc_fund_cd}`。`assoc_fund_cd`は協会ファンドコードで基準価額取得に使用）
- `news_feeds`（市況ニュースの取得元リスト。各要素は`{url, name}`）
- `trading_notes`（投資制度固有の売買特性の自由テキスト。system promptに注入する）

投稿先、間隔、商品、ニュースソース、制度特性のすべてをデータとして持つため、変更にデプロイを要しない。
これはsourcesの`posting_schedule`（自由テキストをエージェントが解釈する）と同じパターンである。
初期データは`advisor_id=ideco-sbi`の1件で、`trading_notes`に「スイッチングは指示から完了まで概ね1週間から10日（レコードキーパー公表の目安）、掛金の配分変更は翌月拠出分から反映」と記載する。
既存の`manage_sources.py`と同パターンの`make advisors-list / advisors-add / advisors-delete`で管理する。

**`advisor-judgments`**：実行ごとの判断履歴。
PKは`advisor_id`、SKは`run_date`。
属性は`judgments`（`{isin, judgment, reason, nav}`のリスト。`judgment`はBUY/SELL/HOLD、`nav`は判断時の基準価額）。
1回の実行 = 1アイテムのため、前回判断の参照も成績検証もPK指定のQuery1回で全履歴を取得できる。

### データソース

- **基準価額**：投資信託協会「投信総合検索ライブラリー」のCSVダウンロードを使う。ISINと協会ファンドコードを指定すれば全ファンドを統一形式で取得でき、日次の基準価額と純資産総額の時系列が得られる。新ツール`tools/nav_fetch.py`として実装する
- **市況ニュースとマクロ情勢**：既存の`rss_fetch`と`web_scrape`を再利用し、アドバイザー定義の`news_feeds`に登録されたフィードから取得する。別角度のニュースソースの追加は`news_feeds`への登録だけで済む
- **為替レート（USD/JPY）**：既存の`api_fetch`を再利用し、認証不要の無料FX API（例: frankfurter.app、ECB公表レート）から現在値と直近の時系列を取得する。新ツールは追加しない。変動見通しはAPIからは得られないため、エージェントが金利差や市況ニュースから定性的に導く

SBI証券サイトのスクレイピングは行わない。
ログインが必要であり、規約上のリスクとサイト変更への脆さがあるためである。

### エージェントと責務分担

数値計算はPython、解釈と判断はLLMに分ける。

- **Python側（handler、純関数）**：アドバイザー定義と履歴の読み書き、冪等性判定、成績の騰落率計算、Slack投稿のオーケストレーション
- **`agent.run_advice`（新設）**：商品リスト、基準価額の時系列、前回判断、成績実績、`trading_notes`、`news_feeds`を入力に、ツール（`nav_fetch`、`rss_fetch`、`web_scrape`）を自律選択して市況を調査し、構造化出力で`{isin, judgment, reason}`のリストを返す

`run_advice`のsystem promptにはアドバイザー共通の規則だけを書き、制度固有の事情は`trading_notes`から注入する。
明記する規則は次のとおり。

- 判定はBUY/SELL/HOLDの3値であり、あいまいな表現で逃げない
- `trading_notes`に記載された売買特性（約定までの日数、反映タイミングなど）を前提とし、その期間内の値動きで無効になる判断をしない。日中や数日の値動きではなく、数週間以上のトレンドと構造的な材料に基づいて判断する
- 前回判断から変更する場合は、何が変わったのかを理由に含める
- USD/JPYの現在値と直近トレンドを必ず取得し、為替ヘッジのない海外資産ファンドの判断では、為替要因と現地資産要因を切り分けて理由を述べる

### Slack投稿形式

1回の実行で1スレッドを投稿する。

- **親メッセージ**：今週の市況サマリ（USD/JPYの現在値と変動見通しを含む）、判断変更点のハイライト、過去推奨の成績サマリ、免責（投資判断情報であり最終判断は利用者が行う）
- **スレッド返信（商品ごと）**：judgmentを絵文字付きで明示（🟢 買い、🔴 売り、🟡 ホールド）し、理由と前回からの変化（例: 前回ホールド→今回売り）を添える

### テスト方針

既存方針を踏襲する。
motoでDynamoDBをモックし、`responses`で投信協会CSVとRSSをモックする。
Bedrockを呼ぶ`run_advice`本体は統合テストせず、`nav_fetch`のCSVパース、騰落率計算の純関数、冪等性判定を含むhandlerの分岐フロー（agentはモック）を検証する。

## Consequences

- リポジトリの責務が「技術ブログdigest」から「情報配信bot全般」に広がる。digestとadvisorはhandlerの分岐以降で独立しており、コード上は相互に影響しない
- 別のアドバイザーの追加は`advisors`へのアイテム追加、ニュースソースの追加は`news_feeds`の変更、制度特性の変更は`trading_notes`の変更で済み、いずれもデプロイを要しない。コード変更が必要になるのは、判断の入力の種類自体が増える場合（新しい取得ツールの追加など）に限られる
- Bedrockの呼び出しコストが増える。週1回、対象10本程度の想定では影響は小さい。`interval_days`を短縮するとコストは比例して増える
- 投信協会のCSVエンドポイントは公式に保証されたAPIではなく、変更されるリスクがある。取得は`nav_fetch`ツールに閉じているため、変更時はツールの差し替えで対応する
- LLMの投資判断の品質は保証されない。成績検証を毎回の投稿に含めることで、判断の信頼性を利用者が継続的に確認できるようにする
- 冪等性判定はSlack履歴に依存するため、botがチャンネルの履歴を読めること（`channels:history`スコープとチャンネル参加）が前提になる。読み取りに失敗した場合は誤って重複投稿するより安全側に倒し、投稿せずエラーで終了する
